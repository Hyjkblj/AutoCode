三 Agent 协同计划（需求 + 实现 + PR）
日期：2026-04-01

说明
- 以 `origin/master` 为进度基线。
- 本文档包含：需求、实现方案、PR 任务清单。

一、需求（做成什么）
1. Python Agent 从规则引擎升级为 LLM 驱动。
- IntentAgent：从关键词匹配升级为 LLM 意图识别（失败回退规则）。
- PlannerAgent：从固定模板升级为 LLM 动态计划（失败回退模板）。
- CoderAgent：从“追加注释”升级为“真实文件改写 + unified diff 输出”。
- ReviewerAgent：从 `REVIEW_BLOCKER` 扫描升级为“基于 diff 的 LLM 审查”。

2. 任务闭环能力必须完整。
- 测试失败触发 Fix Loop，最多 3 次重试。
- 失败耗尽返回 `fix_loop_exhausted`，成功则立即 `TASK_DONE`。
- 多文件修改支持：逐文件处理、逐文件发布 `FILE_PATCH_PREVIEW`。

3. 控制面与移动端要兼容新语义。
- 控制面：兼容 review 风险字段、fix loop 失败原因与审计落库。
- 移动端：展示 review 风险、issues、fix loop 进度与结果。

4. 验收门禁。
- Python：`cd python-agent && pytest -q`
- Java：`mvn -B -q -pl shared-protocol,control-plane-spring,pc-agent-java -am test`

二、如何实现（怎么做）

阶段 P0：先定契约，再改入口
1. 在 `shared-protocol/**` 定义并收敛 LLM 升级事件契约。
2. 在 `python-agent/llm/**` 落地统一 LLMClient（OpenAI/Claude、温度/模型配置、markdown fence 清洗）。
3. 改造 `intent_agent.py` 与 `planner_agent.py` 为 LLM 优先 + fallback。

阶段 P1：打通执行主链路
1. 新增 `python-agent/utils/diff_utils.py`，统一 diff 生成与实质变更判定。
2. 改造 `coder_agent.py`：支持 target_files、多文件串行、真实改写、失败隔离。
3. 改造 `reviewer_agent.py`：基于 diff 的风险分级（high/medium/low）与降级策略。
4. 对齐 `pc-agent-java/**` sandbox 响应契约，保证 Python/Java 语义一致。

阶段 P1/P2：闭环与可视化
1. 改造 `orchestrator/agent_orchestrator.py`：引入 Fix Loop（最多 3 次）并发布迭代事件。
2. 控制面兼容新事件字段并补 E2E 鉴权与闭环测试。
3. 移动端新增 review/fix loop 相关 UI 与状态展示。
