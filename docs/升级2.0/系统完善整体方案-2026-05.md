# AutoCode 系统完善整体方案
更新时间：2026-05-05  
版本：v1.0  
适用范围：`shared-protocol`、`control-plane-spring`、`python-agent`、`pc-agent-java`、`gateway-service`、`artifact-service`、`event-service`、`approval-service`、`ops/*`

---

## 1. 方案目标

本方案用于将当前 AutoCode 从“架构方向正确但仍处于过渡态”的系统，提升为“可稳定演进、可规模化运行、可审计运维”的平台级系统。

核心目标分为四类：

1. 稳定性目标
   - 打通任务创建、执行、事件回传、制品托管、下载访问的主链路稳定性
   - 将关键链路从“可用”提升到“可依赖”

2. 架构目标
   - 收敛当前过渡态架构
   - 明确控制面、执行面、网关、共享协议、微服务边界
   - 降低跨语言、多模块、多运行时带来的系统复杂度

3. 工程目标
   - 建立强契约、强验证、强回归保护体系
   - 让测试、CI、部署、回滚形成一套稳定工程闭环

4. 运营目标
   - 让系统具备面向生产的可观测性、可排障性、可审计性、可恢复性
   - 为后续灰度发布、容量扩展、成本治理做好平台准备

---

## 2. 当前系统判断

从架构师视角，当前系统已经具备平台雏形，优点很明显：

1. 主干架构方向正确
   - 已形成 `Control Plane + Python Agent + Sandbox + Shared Protocol + Gateway + Observability` 主轴

2. 可靠性模式已开始引入
   - 有 ACK、重试、去重、分布式锁、租约、熔断、健康检查等设计基础

3. 平台化意识较强
   - 有共享协议层
   - 有插件与工作流演进方向
   - 有可观测性与灰度思路

4. 工程化投入已经开始形成规模
   - 测试量较大
   - CI、覆盖率、属性测试、集成验证都在逐步补齐

但当前系统的主要问题也很集中：

1. 过渡态过重
   - 单体控制面与拆分服务并存
   - 旧链路与新链路并存
   - 设计上先进，实际运行形态仍较混杂

2. 契约治理不够硬
   - 有共享协议，但跨端仍可能出现响应结构、错误语义、字段含义漂移

3. 运行环境一致性不足
   - 生成、校验、运行、部署之间仍存在环境依赖差异
   - “文档说能跑”和“产物真能跑”之间尚未完全闭环

4. 维护成本在上升
   - Java、Python、Gateway、Sandbox、微服务、工作流、插件多线并进
   - 如果继续扩功能而不收敛边界，复杂度会先于能力增长

---

## 3. 总体策略

本轮系统完善不建议推倒重来，而应采取“收敛式增强”策略。

总体原则如下：

1. 先收敛，再扩张
   - 优先修复跨端契约、主链路闭环、部署一致性
   - 不以继续堆功能为第一优先级

2. 保持增量演进
   - 保留现有模块结构
   - 通过替换主路径、淘汰旧路径，而不是一次性重写

3. 用强契约约束跨语言系统
   - Java 与 Python 的通信边界必须通过共享 schema 和契约测试约束

4. 用运行验证替代文档假设
   - 对“能生成”“能验证”“能部署”“能恢复”全部建立自动化验证

5. 用生产化标准衡量系统成熟度
   - 不只看功能完成度，还要看故障恢复、审计、可观测、灰度与回滚能力

## 4. 目标架构蓝图

目标架构建议稳定为以下五层：

### 4.1 协议层

职责：

1. 定义所有跨服务、跨语言通信契约
2. 管理 schema、示例、兼容性规则、版本策略
3. 提供 Java/Python 双侧契约校验能力

建议：

1. `shared-protocol` 作为唯一协议源
2. ACK、事件、制品元数据、沙箱协议全部版本化
3. 所有服务不得自行发明新的临时字段格式

### 4.2 控制层

职责：

1. 任务生命周期管理
2. 事件接收与状态折叠
3. 审批、安全、审计、项目级授权
4. 控制面统一查询接口

建议：

1. `control-plane-spring` 在中期内继续承担主控制中枢角色
2. 只保留“编排控制与策略”职责
3. 将与制品托管、事件处理、审批执行高度耦合的具体实现逐步外提到独立服务

### 4.3 执行层

职责：

1. 意图识别、计划生成、代码生成、校验修复、结果发布
2. Outbox、重试、熔断、资源限制、执行上下文管理
3. LangGraph 与 legacy engine 的统一调度入口

建议：

1. `python-agent` 作为统一执行运行时
2. 强化“工作流内核”与“生成器/插件”分离
3. 把环境依赖、运行时依赖、生成物依赖明确分层

### 4.4 接入层

职责：

1. 网关路由
2. 认证头传播
3. 限流、超时、错误整形
4. 统一入口与外部可观测指标暴露

建议：

1. `gateway-service` 成为统一入口
2. 外部访问不直接穿透到各内部服务
3. 逐步把 nginx 临时逻辑迁移为网关策略或统一基础设施能力

### 4.5 运行保障层

职责：

1. 指标、日志、链路追踪
2. 告警、巡检、烟测、回滚
3. 灰度、金丝雀、容量与成本治理

建议：

1. Prometheus + Grafana + Alertmanager 保持主方案
2. 所有关键主链路都要有对应指标与告警
3. 每个阶段升级都必须带 smoke test 与 rollback 手册

## 5. 系统完善的六大工作流

### 5.1 工作流一：主链路稳定化

目标：让系统“真正能稳定跑通”

范围：

1. 任务创建 -> 任务轮询 -> 任务执行 -> 事件回传 -> 状态更新 -> 制品上传 -> 制品访问
2. ACK 语义、错误语义、重试语义闭环
3. 失败后可恢复、重复事件不污染状态

重点动作：

1. 收敛 ACK 协议
   - 控制面只输出一种响应封装
   - Python 客户端只解析一种响应结构
   - 非重试错误统一归类

2. 收敛事件模型
   - 明确事件 `eventId`、`seq`、`accepted`、`duplicate`、`errorCode` 的唯一语义
   - 明确控制面序号与 agent 侧序号是否同义

3. 强化任务状态机
   - 定义合法状态迁移表
   - 对非法事件直接拒绝并审计

4. 强化制品主路径
   - 上传、列举、下载、托管 URL 使用统一服务边界

成功标准：

1. 主链路 smoke test 成功率 >= 99%
2. 重复事件不导致状态污染
3. 非重试错误不会触发指数退避重试

### 5.2 工作流二：契约与兼容性治理

目标：避免 Java/Python 跨端漂移

范围：

1. API 响应结构
2. 事件 schema
3. 共享 DTO/模型语义
4. 示例与校验器一致性

重点动作：

1. 建立“协议变更四件套”
   - schema
   - example
   - Java validator
   - Python validator

2. 增加契约测试层
   - 控制面返回的关键响应由 Python 侧回放验证
   - Python 侧生成的事件由 Java 侧 validator 校验

3. 建立兼容策略
   - 向后兼容字段只增不删
   - 破坏性升级必须走版本号

成功标准：

1. 关键跨端接口全部有 schema 和 example
2. 每次协议变更自动触发契约测试
3. 不再出现“编译通过但协议失配”的问题

### 5.3 工作流三：执行与生成体系收敛

目标：让 `python-agent` 从“功能集合”演进为“稳定执行内核”

范围：

1. BackendGenerator / FastAPIGenerator / FullstackGenerator
2. ValidationGate / FixLoop
3. LangGraph workflows
4. Plugin runtime

重点动作：

1. 明确三层依赖
   - agent 运行时依赖
   - 生成产物依赖
   - 校验环境依赖

2. 统一校验门
   - backend 和 fullstack workflow 全部复用同一 `ValidationGate`
   - 校验规则按 target 维度统一管理

3. 收敛生成器输出标准
   - backend/fullstack 生成结果必须满足最小运行契约
   - 所有 README、compose、Dockerfile、nginx 配置必须与实际输出一致

4. FixLoop 只处理代码缺陷，不吞环境缺陷
   - 缺依赖、缺工具、环境不一致应归类为平台问题

成功标准：

1. backend/fullstack 生成成功后可自动完成最小运行验证
2. 生成物的文档、依赖、部署模板一致
3. workflow 不再只做弱校验

### 5.4 工作流四：服务边界重构

目标：把“设计上的微服务”变成“职责清晰的可维护系统”

建议的中期边界：

1. `control-plane-spring`
   - 负责任务、状态、策略、聚合查询、审计入口

2. `event-service`
   - 负责事件接收、去重、持久化、ACK 生成
   - 如果短期不真正独立，则至少先实现逻辑边界独立

3. `artifact-service`
   - 负责制品存储、下载、托管与生命周期

4. `approval-service`
   - 负责审批流、权限判定、审批审计

5. `gateway-service`
   - 负责统一外部入口和外部访问策略

边界原则：

1. 数据所有权唯一
2. 写路径尽量单点归属
3. 查询可以聚合，写入不可多头
4. 先做逻辑边界，再做物理拆分

成功标准：

1. 每个服务职责一句话能说清楚
2. 不再存在“一项业务同时在两个服务里都能写”的情况
3. 控制面代码复杂度逐步下降

### 5.5 工作流五：生产可运维化

目标：让系统在生产环境可观测、可定位、可恢复

重点动作：

1. 指标体系
   - 任务吞吐、任务成功率、重试次数、ACK 失败率、生成成功率、校验失败分类、制品下载成功率

2. 日志体系
   - 统一字段：`taskId`、`traceId`、`runId`、`stage`、`errorCode`、`eventType`

3. 告警体系
   - 事件失败率
   - 长时间积压
   - 生成成功率下降
   - 健康检查失败
   - 制品服务不可达

4. 运行手册
   - 启动手册
   - 烟测手册
   - 故障排查手册
   - 回滚手册

成功标准：

1. 任一关键故障可在 10 分钟内定位到模块
2. 升级发布有标准 smoke test
3. 回滚路径清晰、自动化程度可接受

### 5.6 工作流六：工程化与治理体系

目标：让系统具备持续迭代能力

重点动作：

1. 测试分层
   - 单元测试
   - 契约测试
   - 组件集成测试
   - 端到端 smoke test
   - 属性测试

2. CI 收敛
   - 关键服务编译
   - 关键 Python 文件语法校验
   - 契约测试
   - 最小 smoke test
   - 覆盖率门禁

3. 代码治理
   - 明确过渡代码淘汰清单
   - 清理未接线的模板与临时逻辑
   - 明确“实验特性”和“生产主链路”的隔离边界

4. 架构治理
   - 重大变化必须有 ADR
   - 协议变化必须有 schema/example/test
   - 服务拆分必须有边界定义和回滚方案

成功标准：

1. 关键改动无法绕过自动验证
2. 试验性能力不会污染主链路
3. 技术债有清单、有优先级、有退出计划

1. ACK 协议统一

1. 架构综合分从 `7.4` 提升到 `7.8`

1. 服务边界图
2. 数据所有权清单
3. 旧路径下线计划

阶段评分目标：

1. 服务边界清晰度提升到 `8.0`

### 阶段 C：生产成熟化期

阶段评分目标：

## 8. 组织与协作建议

建议把工作按四类 owner 划分：

1. 架构与协议 owner
   - 负责共享协议、服务边界、ADR、兼容性策略

2. 控制面 owner
   - 负责任务状态机、事件接入、权限审批、审计

3. 执行与生成 owner
   - 负责 Python agent、workflow、generator、validation、fix loop

4. 平台运维 owner
   - 负责网关、部署、观测、告警、灰度、回滚

协作机制建议：

1. 每周一次架构收敛例会
2. 每个阶段有明确退出条件
3. 重大跨端改动必须先改协议，再改实现，再补回归测试

## 9. 风险与对策

### 风险一：继续扩功能导致收敛失败

表现：

1. 模块越来越多
2. 过渡逻辑越来越厚
3. 修复成本越来越高

对策：

1. 阶段 A 和 B 期间限制新增大功能
2. 所有新增能力必须证明不扩大主链路复杂度

### 风险二：微服务拆分过早

表现：

1. 物理拆分先于职责拆分
2. 远程调用增加但边界仍不清

对策：

1. 先做逻辑边界，再做物理拆分
2. 先统一协议和数据所有权，再推广独立部署

### 风险三：测试很多但关键问题仍漏出

表现：

1. 编译通过
2. 测试通过
3. 线上仍有协议失配或环境失配

对策：

1. 增加契约测试和 smoke test
2. 增加“生成物可运行性”验证

### 风险四：运行环境不一致

表现：

1. 本地能跑
2. CI 不稳定
3. 部署后产物不可运行

对策：

1. 固化 agent 运行时依赖
2. 固化校验环境
3. 固化生成物最小部署模板

## 11. 一句话结论

AutoCode 当前最需要的不是继续横向加能力，而是以“协议收敛、主链路稳定、边界清晰、生产化闭环”为核心，完成一次系统性的架构收口。  
只要这轮收口做好，现有架构完全有机会从“过渡态平台”成长为“稳定可扩展的平台型系统”。
