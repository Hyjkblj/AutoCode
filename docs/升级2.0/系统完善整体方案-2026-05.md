# AutoCode 系统完善整体方案
更新时间：2026-05-05  
版本：v1.2  
适用范围：`shared-protocol`、`control-plane-spring`、`python-agent`、`pc-agent-java`、`mobile-app`、`ops/*`

---

## 1. 方案定位

本方案不是一份“把当前系统立刻改造成微服务平台”的蓝图，而是一份基于当前 README 所呈现系统现实形态的收敛方案。

当前 AutoCode 更准确的判断，不是“混乱的过渡态微服务系统”，而是：

1. 一个以 `control-plane-spring` 为核心的单体控制面
2. 两个职责明确的协作节点：`python-agent` 与 `pc-agent-java`
3. 一个跨语言协议层：`shared-protocol`
4. 若干已经开始出现、但尚未必须独立拆分的生产化诉求

因此，本方案的目标不是优先拆服务，而是优先把这套“单体控制面 + 协作式 Agent”架构打磨到稳定、清晰、可运维、可演进。

---

## 2. 总体判断

从 README 视角看，当前系统已经具备相当清晰的基础骨架：

1. 控制面职责集中
   - `control-plane-spring` 统一负责任务生命周期、状态机、权限、审批、产物管理与广播

2. Agent 分工明确
   - `python-agent` 负责 AI 编排、代码生成、校验与工作流
   - `pc-agent-java` 负责沙箱执行、安全门控、命令策略与本地执行约束

3. 交互边界基本清楚
   - 控制面与 Agent 通过 API 与事件协作
   - Java/Python 通过共享协议定义核心 DTO 与事件契约

4. 当前最大的风险不在“模块太少”，而在“真实运行约束还没有被工程化收敛”

所以，本方案的核心前提调整为：

1. 当前主架构应视为“有效的 V1 单体控制面 + 协作式 Agent 架构”
2. 不把“未拆微服务”本身当成问题
3. 只有在吞吐、团队协作、部署独立性和故障域已经证明单体成为瓶颈时，才进入物理拆分议程

## 3. 当前系统的真实问题

相比“是否拆服务”，当前更真实、更高优先级的问题有以下六类。

### 3.1 主链路稳定性仍需收口

包括：

1. 任务轮询、事件 ACK、重试、去重、状态折叠的语义闭环
2. 生成产物是否真的可运行、可部署、可访问
3. 非重试错误是否会被误重试

这里的问题是“链路完整性”，不是“服务数量不足”。

### 3.2 双认证模式带来的安全与维护债务

README 已明确表明当前控制面同时存在：

1. `token` 模式
2. `jwt` 模式
3. `jwt` 模式下对 `X-Agent-Token` 的兼容适配

这意味着系统当前实际上存在两套认证路径与兼容逻辑。

风险包括：

1. 安全策略分叉
2. 代码路径分叉
3. 调试难度上升
4. 后续权限模型收敛困难

这是一项真实且高优先级的技术债。

### 3.3 沙箱网络模型带有明显单机假设

README 已暴露以下关键事实：

1. 沙箱服务仅监听 `127.0.0.1`
2. `python-agent` 与 `pc-agent-java` 共享网络命名空间
3. 调用路径依赖本地回环地址

这是一套非常务实、也非常有效的单机部署模型，但它同时意味着：

1. 当前系统默认部署边界是“同机协作”
2. 后续如需跨主机扩展 Agent，将面临根本性的网络模型调整
3. 这不是小优化，而是架构约束

因此，这一问题必须被明确建档，而不是被笼统归入“后续平台化”。

### 3.4 制品生命周期策略不清晰

README 明确提到：

1. 产物会被打包
2. 用户通过短链接访问生成结果
3. `python-agent` 可以打包上传 `export.zip`

但仍缺少一套明确、可执行的制品治理策略：

1. 产物最终落在哪里
2. 本地磁盘、共享卷、对象存储的边界如何定义
3. `export.zip` 的生命周期、清理策略、失败回收策略是什么
4. 托管内容与原始归档之间的关系是什么

这类问题往往不是架构图上最显眼的，但最容易在生产里造成磁盘膨胀、脏数据和访问异常。

### 3.5 AI 工作流的调试与可观测性不足

对 AutoCode 这种系统来说，真正难排查的问题通常不是 CPU 或内存，而是：

1. 为什么这个 prompt 生成出了错误结果
2. 哪一个 Agent 阶段引入了偏差
3. 哪次 LLM 调用开始明显退化
4. LangGraph 或 DAG 流程在什么节点决策失真

因此，对 `python-agent` 来说，最有价值的可观测性并不是先上齐所有通用指标，而是优先补：

1. prompt / response 追踪
2. token 消耗与时延
3. workflow 节点级状态可视化
4. 生成失败分类

### 3.6 契约问题的重点在业务语义，而不只是字段结构

`shared-protocol` 已经存在，这意味着当前并不是“完全无契约”的状态。

真正更危险的是业务语义漂移，比如：

1. `ARTIFACT_READY` 到底表示“本地已生成”还是“控制面已可访问”
2. ACK 中的 `seq` 到底是谁的序号
3. 某类失败到底该由 Agent 重试还是控制面拒绝

这类问题不能仅靠 schema 解决，必须通过业务语义文档、端到端冒烟测试和关键流程约定来约束。

## 4. 修正后的总体策略

本轮系统完善采用“单体收敛优先、真实约束优先、证据驱动演进”的策略。

### 4.1 基本原则

1. 先修屋顶，再谈加盖
   - 优先解决真实会导致生产不稳定的问题

2. 单体控制面优先保留
   - 在没有明确证据前，不推动 `control-plane-spring` 物理拆分

3. 逻辑边界可以增强，物理边界延后决策
   - 可以整理模块边界、包结构、接口隔离
   - 但不预设一定要拆成多个独立服务

4. Java Agent 视为协作节点，而非从属执行模块
   - 保持其在安全与本地策略上的自治性

5. 契约治理强调“高价值少数接口”
   - 先抓最关键主链路，而不是全面铺开重量级治理

6. 可观测性优先服务排障
   - 先解决“出了问题怎么定位”，再扩展到“全平台仪表盘完备化”

## 5. 校准后的目标架构

未来一段时间内，建议将目标架构稳定理解为四个核心域，而不是急于定义多个独立微服务。

### 5.1 控制域：`control-plane-spring`

职责：

1. 任务生命周期与状态机
2. 任务分发与事件接入
3. 权限、审批、审计
4. 产物元数据、托管入口与访问控制

建议：

1. 继续作为单体控制核心
2. 内部按领域整理逻辑边界
3. 对事件、产物、审批做包级或模块级隔离
4. 仅在证据明确时才考虑物理拆分

### 5.2 AI 编排域：`python-agent`

职责：

1. 意图识别
2. 计划生成
3. 代码生成
4. 校验与修复
5. 结果打包与上报

建议：

1. 作为主要的 AI 工作流运行时
2. 优先解决工作流可观测性、生成闭环验证、依赖分层问题
3. 先收敛 legacy 与 LangGraph 的主路径边界，再逐步扩能力

### 5.3 执行与安全域：`pc-agent-java`

职责：

1. 命令执行节点运行时
2. 本地沙箱 API
3. 策略校验与高风险门控
4. 审批绑定与本地安全边界

建议：

1. 作为独立协作节点持续保留
2. 明确其“节点自治”定位
3. 把本地网络假设、审批绑定、安全策略版本化管理

### 5.4 协议域：`shared-protocol`

职责：

1. 共享 DTO 与事件结构
2. Java/Python 之间的最小公共语义
3. 关键协议的示例与约束

建议：

1. 保持为唯一协议源
2. 只对关键高风险接口强化治理
3. 不以“文档齐全”为目标，而以“关键链路不漂移”为目标

## 6. 真正高优先级的六个工作流

### 6.1 工作流一：主链路闭环与运行一致性

目标：

1. 让任务主链路真正稳定可依赖
2. 让生成物不是“看起来完整”，而是真的可运行

范围：

1. 任务创建 -> 轮询 -> 执行 -> 事件回传 -> 状态更新 -> 产物上传 -> 访问
2. fullstack/backend 生成结果的最小运行验证
3. 部署模板与生成结果一致性

重点动作：

1. ACK 语义统一
2. 非重试错误显式归类
3. fullstack/backend 的 ValidationGate 统一收敛
4. README / compose / Dockerfile / nginx 配置与实际输出对齐

成功标准：

1. 主链路 smoke test 稳定通过
2. backend/fullstack 产物具备最小运行闭环
3. “文档可运行”与“真实可运行”不再脱节

### 6.2 工作流二：认证模式收敛

目标：

1. 降低双认证模式带来的安全债与维护债

范围：

1. `token`
2. `jwt`
3. `X-Agent-Token` 兼容适配路径

重点动作：

1. 盘点当前认证入口与校验路径
2. 明确“用户认证”和“Agent 认证”的边界
3. 设计单一主认证模型
4. 为兼容路径设置退场计划

认证收敛可选方向与取舍：

1. 方向 A：以 `JWT` 作为控制面主认证，`X-Agent-Token` 仅保留为 Agent 兼容适配层
   - 优点：与当前 README 描述最贴近，收敛成本最低，便于先清理双路径判断
   - 缺点：兼容层在退场前仍需维护一段时间
   - 适用阶段：当前阶段的首选路线

2. 方向 B：用户侧统一 `JWT`，Agent 侧切换为专用服务凭证或签名 Token
   - 优点：可把“用户身份”与“节点身份”彻底分离，便于后续跨主机扩展
   - 缺点：需要新增节点凭证签发、轮换与吊销机制，实施成本高于方向 A
   - 适用阶段：当 Agent 数量增加、节点治理复杂度明显上升时纳入正式设计

3. 方向 C：跨主机高安全场景采用 `mTLS + 节点凭证`
   - 优点：更适合多主机、强隔离、零信任网络场景
   - 缺点：证书生命周期、部署复杂度、运维门槛都会显著上升
   - 适用阶段：仅在跨主机部署成为明确需求后评估，不作为当前收敛首选

当前建议：

1. 先按方向 A 收敛
2. 同时把方向 B 作为下一阶段候选演进路线写入 ADR
3. 不在当前版本直接引入方向 C

成功标准：

1. 控制面认证主路径清晰且单一
2. Agent 兼容适配器有明确保留范围与退场条件
3. 安全规则不再依赖多套并行判断
4. 至少形成一页《认证收敛可选方向与取舍》说明，避免实施阶段重复争论

### 6.3 工作流三：沙箱网络模型与执行边界治理

目标：

1. 把当前“同机部署假设”从隐含前提变成显式约束
2. 为未来跨主机扩展预留演进路径

重点动作：

1. 明确当前支持的部署拓扑
2. 把 `127.0.0.1`、共享网络命名空间、端口假设写入运行文档
3. 抽象 Agent -> Sandbox 的连接配置
4. 设计跨主机模式的后续演进草案，但不立即实施

启动跨主机网络模型设计的触发条件：

1. 需要把 `python-agent` 或 `pc-agent-java` 部署到独立主机
2. 同机模式下的资源争用已影响任务成功率、延迟或隔离要求
3. 安全合规要求 Agent 与 Sandbox 运行在不同主机或网段
4. 单机网络命名空间假设已经成为新部署拓扑的阻塞项

成功标准：

1. 当前网络模型被明确记录
2. 部署失败时能快速定位是否属于拓扑不匹配
3. 后续扩容时不需要重新理解底层假设
4. 跨主机演进草案有明确启动信号，而非无限期搁置

### 6.4 工作流四：制品存储与生命周期治理

目标：

1. 让制品系统具备真正可运维性

重点动作：

1. 明确制品存储分层
   - 原始归档
   - 托管静态内容
   - 元数据索引

2. 明确生命周期策略
   - 创建
   - 上传
   - 托管
   - 过期
   - 清理

3. 明确失败回收机制
   - 上传中断
   - 托管失败
   - 元数据与物理文件不一致

4. 明确实现路线
   - 先支持当前单体控制面下的稳定存储模型
   - 是否上对象存储由容量与部署需求决定

成功标准：

1. 产物的落盘、访问、清理路径都能说清楚
2. 不再出现“文件还在但元数据没了”或“元数据还在但文件丢了”的暗区
3. 定时清理与过期治理可自动执行

### 6.5 工作流五：AI 专项可观测性

目标：

1. 优先解决 AI 工作流难排障的问题

重点动作：

1. 为 `python-agent` 增加节点级追踪
2. 记录关键 LLM 调用的：
   - model
   - latency
   - token usage
   - prompt class
   - failure type

3. 区分失败分类：
   - 模型调用失败
   - 生成内容不合格
   - 校验失败
   - 环境依赖失败
   - 沙箱执行失败

4. 为 LangGraph / DAG 运行提供最小可视化状态
5. 对 `mobile-app` 维持轻量补充策略
   - 当前阶段沿用现有基础监控
   - 仅补充与主链路直接相关的指标，如 WebSocket 连接稳定性、任务创建延迟、语音输入到任务落库时延
   - 待核心链路稳定后，再决定是否建设移动端专项观测面板

成功标准：

1. “为什么生成失败”可快速归因到具体阶段
2. 模型退化、成本异常、节点阻塞可观测
3. 排障依赖从“猜测”转向“证据”
4. 移动端仅覆盖主链路相关基础观测，不提前引入重型专项体系

### 6.6 工作流六：轻量高价值契约治理

目标：

1. 用合适强度的治理控制跨端漂移

重点动作：

1. 只优先覆盖高价值接口：
   - 任务轮询响应
   - 事件 ACK 响应
   - 沙箱执行请求/响应
   - 产物元数据

2. 对关键事件补齐语义文档
3. 契约测试与端到端 smoke test 结合使用
4. schema 主要解决结构问题，业务文档解决语义问题

成功标准：

1. 关键接口有明确语义
2. Java/Python 对关键响应的理解保持一致
3. 不把契约治理扩展为低收益的全量文档工程

## 7. 对“服务拆分”的修正立场

服务拆分不再作为当前阶段的默认主线，而是作为一个有前置条件的中期选项。

### 7.1 当前立场

1. `control-plane-spring` 暂不进行物理拆分
2. 允许在代码层做领域内聚和边界清理
3. 只在以下条件同时出现时，才考虑物理拆分：
   - 团队边界已经独立
   - 发布节奏已互相牵制
   - 故障域隔离已成为硬需求
   - 数据与事务边界已经足够明确

### 7.2 当前可做的事情

1. 画出逻辑领域边界
2. 明确数据所有权
3. 整理事件、审批、制品的代码结构
4. 为未来拆分保留边界，但不预支分布式复杂度

### 7.3 当前不建议做的事情

1. 没有业务证据时预先拆 `event-service`
2. 没有存储策略收敛时预先拆 `artifact-service`
3. 没有授权模型收敛时预先拆 `approval-service`

## 8. 分阶段实施路线

### 阶段 A：止血与收口期

周期建议：2-3 周

目标：

1. 修主链路
2. 修运行一致性
3. 明确最关键真实约束

关键产出：

1. ACK 与重试语义收敛
2. fullstack/backend 生成物最小运行闭环
3. 制品路径与部署模板对齐
4. 沙箱网络模型文档化

### 阶段 B：技术债治理期

周期建议：3-4 周

目标：

1. 收敛认证模式
2. 建立制品生命周期治理
3. 补齐 AI 专项可观测性

关键产出：

1. 认证主路径方案
2. 兼容路径退场计划
3. 制品存储与清理策略
4. Python Agent 节点级追踪能力
5. 《认证收敛可选方向与取舍》说明页

### 阶段 C：结构清晰化期

周期建议：3-5 周

目标：

1. 清理控制面内部逻辑边界
2. 强化高价值协议治理
3. 建立端到端稳定验证体系

关键产出：

1. 控制面内部领域边界图
2. 关键接口语义说明
3. 高价值契约测试 + smoke test 组合方案

### 阶段 D：证据驱动的平台演进期

周期建议：持续进行

目标：

1. 根据真实增长判断是否需要拆服务
2. 推进 LangGraph、插件能力和更多框架支持
3. 持续优化性能、成本与恢复能力

关键产出：

1. 是否拆服务的判定清单
2. 跨主机 Agent 演进草案
3. 成本与容量治理方案
4. 跨主机网络模型设计触发条件清单

## 9. 里程碑

### M1：主链路可信

完成标准：

1. 关键 ACK 问题清零
2. 生成物部署闭环可验证
3. smoke test 稳定

### M2：真实技术债可治理

完成标准：

1. 双认证模式收敛路径明确
2. 制品生命周期策略落地
3. 沙箱拓扑约束清晰
4. 认证收敛备选路线与取舍已文档化

### M3：AI 排障能力成形

完成标准：

1. Python Agent 关键工作流可追踪
2. 失败归因可定位到节点与类型
3. 移动端主链路观测边界已明确

### M4：是否拆服务有证据可依

完成标准：

1. 控制面内部逻辑边界清晰
2. 具备明确的拆分前置条件评估
3. 拆与不拆都能基于证据而非想象决策

---

## 10. 结论

对当前 AutoCode 来说，真正正确的下一步不是优先把控制面拆成更多服务，而是先把当前这套“单体控制面 + 协作式 Agent + 共享协议”的架构磨成熟。

优先级应明确为：

1. 主链路闭环
2. 双认证收敛
3. 沙箱网络模型显式化
4. 制品生命周期治理
5. AI 专项可观测性
6. 高价值少数接口的契约治理

只有当这些问题被解决，并且真实业务增长已经证明单体控制面成为瓶颈时，服务拆分才是一个值得投入的下一阶段动作。
