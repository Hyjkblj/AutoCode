# MVP 版本可行性测试方案（MobileVoice-CodeOps）

## 1. 目标
本方案用于在 **2 周内**验证项目 MVP 是否可行，输出明确结论：`GO / CONDITIONAL GO / NO-GO`。

验证问题：
1. 语音输入是否能稳定转成可执行任务。
2. Java 控制平面是否能完成任务编排、实时事件推送、审批流。
3. PC Node Agent 是否能稳定执行助手任务并回传结构化事件。
4. 在弱网/断线下，任务状态和事件流是否可恢复。
5. 当前技术栈（Android Java + Spring Boot + MySQL + Redis + Java Agent）是否满足 MVP 目标。

---

## 2. MVP 范围（本次必须验证）

## 2.1 In Scope
1. Android 端：语音输入、任务创建、任务事件实时展示、审批按钮。
2. 控制平面：`POST /tasks`、`/ws` 事件推送、审批接口、任务状态机。
3. Node Agent：注册/心跳、接单执行、结构化事件回传。
4. 至少一个助手链路可用（建议 Codex）。
5. 高风险动作审批（exec 或 git push 任一）。

## 2.2 Out of Scope
1. 多租户企业权限全量模型。
2. iOS 客户端。
3. Claude 多模型路由优化（可做预研，不纳入 MVP 通过条件）。

---

## 3. 通过门槛（硬标准）

以下 8 项全部达标才判定 `GO`：
1. E2E 成功率 >= `90%`（同一批 30 条任务）。
2. 任务创建 API P95 < `300ms`。
3. 事件推送端到端 P95 < `800ms`。
4. WebSocket 断线重连后可续流（`lastEventId`）。
5. 审批流可阻断并恢复任务执行。
6. 审计日志可追溯（审批人、动作、时间、上下文）。
7. Node 心跳离线检测在 `30s` 内生效。
8. 无 P0/P1 级安全问题（目录逃逸、无审批高风险执行、未鉴权访问）。

若第 1~7 项达标但存在 1~2 个可修复 P1 风险，判定 `CONDITIONAL GO`。

---

## 4. 测试环境基线

1. Android：API 29+ 真机 1 台，模拟器 1 台。
2. 后端：Java 21 + Spring Boot 3.5.x。
3. 数据库：MySQL 8.4 LTS。
4. 缓存：Redis 7。
5. Node Agent：Java 21（Windows 或 Linux 任一）。
6. 网络：正常网络 + 人工弱网（限速/丢包）场景。

建议最小部署：
1. `control-plane` 1 实例
2. `mysql` 1 实例
3. `redis` 1 实例
4. `pc-agent` 1 实例
5. `android-app` 1 客户端

---

## 5. 测试数据准备

1. 任务语料 30 条：
- 10 条低风险（读取/分析）
- 10 条中风险（文件修改）
- 10 条高风险（命令执行/网络写）

2. 项目仓库准备：
- 小型 Java demo 仓库 1 个（用于真实改动）
- 只读仓库 1 个（用于权限校验）

3. 用户角色：
- `Admin` 1 个
- `Operator` 1 个
- `Viewer` 1 个

## 7. 执行步骤（按天）

## Day 7-8：安全验证

1. 全量回归。

## 9. 缺陷分级与处置

1. `P0`：安全绕过/数据破坏/核心链路不可用 -> 立即阻断上线。
2. `P1`：主流程高频失败/状态错乱 -> 修复后复测。
3. `P2`：非主流程异常或体验问题 -> 记录并排期。

发布门禁：
1. `P0=0`
2. `P1<=2` 且有明确修复计划与时间
3. 所有硬性通过门槛满足

---

## 10. Go / No-Go 判定规则

1. `GO`
- 硬门槛全部达标
- 无 P0/P1 阻断缺陷

2. `CONDITIONAL GO`
- 指标基本达标
- 存在少量可控 P1，且 1 周内可修复

3. `NO-GO`
- 任一核心链路失败（语音->执行->回传->审批）
- 或存在 P0 安全问题
- 或性能明显不达标且无短期优化路径

## 14. 当前建议结论（基于架构评审，非实测）

在你当前技术方案下（Java 控制平面 + Java Agent + Android Java + MySQL + Redis），
**MVP 高概率可行，建议先以 Codex 单助手链路做封闭测试。**

进入正式 `GO` 前，必须完成本文件定义的 12 天验证并达到硬门槛。
